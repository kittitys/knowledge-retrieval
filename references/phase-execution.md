# Phase 执行细节

> 本文档是 `SKILL.md` 的详细扩展，覆盖 Phase 0 / A / B 的完整执行步骤。
> 仅在执行对应阶段时读取。

---

## Phase 0 — 搜索环境就绪检查

> 每次搜索前执行。三步筛选，全部通过才进入 Phase A。

### 0.1 路径验证：原始文件夹还在吗？

**方式：** 读取 `data_structure.md` 中 `实际位置` 记录的路径，检查是否存在。

→ **存在** → 进入 0.2
→ **不存在** → 执行路径恢复流程：
  1. 从已知根路径出发，搜索可能的候选位置
  2. 上报用户确认
  3. 更新 data_structure.md 中的 `实际位置`
  4. 进入 0.2

### 0.2 文件索引新鲜度检查

**方式：** `dir /b`（Mac: `ls`）扫描原始文件夹下的文件名列表，对比 data_structure.md 的「文件索引」表。

→ **无差异** → 进入 0.3
→ **有差异** → 自动处理：
  - 新增文件 → 补条目到 data_structure.md（初始描述来自文件名+目录）
  - 已删文件 → 移除条目 + 清对应缓存
  - 标记 BM25 状态为 `stale` → 进入 0.3

> 毫秒级操作，只读文件名，不读文件内容。

### 0.3 索引重建：BM25 需要刷新吗？

→ BM25 索引最新 → 进入 Phase A
→ 标记 `stale` → 执行：
```bash
python .agents/skills/knowledge-retrieval/scripts/build_kb_index.py --project <文件夹名>
```
→ 告知用户「已重建搜索索引」→ 进入 Phase A
→ BM25 索引缺失/损坏 → `search_kb.py` 调用时会自动触发重建（代码级兜底，由 `scripts/search_kb.py` 内的 auto-rebuild 逻辑实现）
→ BM25 未安装 → 告知用户「未检测到 BM25 搜索库，是否安装？如不安装将使用纯 AI 语义匹配」
   → 用户确认 → 优先运行 `scripts/setup.bat`（自动检测 Python + 安装全部依赖）
     → setup.bat 可用 → 安装完成 → 重建索引 → 进入 Phase A
     → setup.bat 不可用（非 Windows/无权限等）→ 备用：`pip install bm25s==0.3.8` → 重建索引 → 进入 Phase A
   → 用户拒绝 → 跳过 BM25 通道，仅用 AI 语义匹配 → 进入 Phase A

## Phase A — 定位目标文件（双通道候选生成）

> 目的：从知识库所有文件中，选出用户问题最相关的候选文件列表。
> Phase A 不负责回答用户问题，只负责回答「该读哪些文件」。

### 通道①：AI 语义匹配

1. 读取 `data_structure.md` 的「描述」列
2. 用自然语言语义判断哪些文件的描述与用户问题相关
3. 含义清晰、匹配明确 → 锁定文件 → 加入候选列表 A
4. 模糊/不确定/零命中 → **不阻止算法通道运行**，由算法通道补充

### 通道②：BM25 算法搜索

**前提：** BM25 索引已构建（Phase 0.3 已保证）。

**Step 1 — LLM 查询扩展：**
将用户问题扩展为搜索词集合（同义词、上下位、英文缩写，≤ 20 词）。
注意力集中在「文件可能使用的词汇」而非「聊天回复用语」。

**Step 2 — 执行 BM25 搜索：**
```bash
python .agents/skills/knowledge-retrieval/scripts/search_kb.py <文件夹名> "<扩展后的搜索词>" --top-k 15
```

**Step 3 — 后处理：**
- 过滤低分条目
- 保留 Top 10 作为候选列表 B

### ③ 合并策略

```
合并 A + B → 去重 → 按以下优先级排序：
  1. AI 语义匹配的文件（AI 有明确判断）
  2. BM25 分数 > 1.0 的文件（强信号）
  3. BM25 分数 > 0.5 的文件（中等信号）
  → 输出 Top 10 候选 → Phase B
  → 零结果 → 承认局限（遵循反幻觉铁律）
```

## Phase B — 阅读候选文件 + 综合回答 + 描述进化

> 输入：候选文件列表（Top 10，按相关性排序）
> 输出：综合理解后的回答

### 模式选择

**模式 1 — 正常模式（默认）：**
AI 逐个读取候选文件，综合理解后回答。现代 LLM 对 5-10 个中短文件的全文阅读+综合有足够成熟的应对能力。

**模式 2 — 大文件保护模式（条件触发）：**
当任一候选文件 > 5 万字，或所有候选累计占用 > 上下文窗口 70% 时：
→ 切换为「关键词搜索 → 命中段落阅读」策略
→ 先用 grep/Select-String 定位相关段落，只读匹配行及前后文

### 文件读取顺序

1. 按相关性排序从高到低逐个读取
2. 读取方式取决于文件类型（见 `file-handling.md`）
3. 每读完一个文件，检查是否已收集到足够的信息回答用户问题
4. 已足够 → 提前退出，不需要读完所有候选

### 去重预处理（多文件时启用）

当候选列表包含 ≥ 2 个文件时，在逐个读取**之前**先做页面级去重：

**流程：**
1. 对每个候选文件，按页提取文字
2. 对每页文字做归一化处理：去除首尾空白、统一换行符为 `\n`、合并连续空白为单空格
3. 对归一化后的全文计算 MD5 哈希
4. 跨文件比对：如果文件 A 第 5 页的哈希与文件 B 第 20 页的哈希匹配 → 判定为重复页
5. 阅读时跳过重复页，并在回答中标注：「文件 B 第 20-25 页与文件 A 第 5-10 页内容一致（已省略）」

**边界条件：**
- 同一文件内部不会自我去重（预设各页不同）
- 仅 ≥ 2 个文件时触发，单文件不处理
- MD5 前务必归一化，否则不同工具提取的同一页可能因格式差异而哈希不匹配
- **PPTX：** 按页去重效果最好（每页是独立 slide，边界清晰）
- **PDF：** 因排版差异，逐页比对可能漏匹配，但无负面影响。章节级去重待后续版本

### 综合回答 + 溯源锚点（强制执行）

**每引用一个文件中的具体信息时，必须附带来源锚点：**
> `[来源: <文件名>#<页码/段落>]`

格式要求：
- 每个关键事实/数据/论点后标注来源
- 多个信息点来自不同文件 → 各自标注
- 同一段话内不同句子的来源可能不同 → 逐句标注
- 无具体来源的 AI 推理 → 不编造来源
- **来源必须指向 data_structure.md 中存在的实际文件名。** 禁止使用「对比分析」「综合判断」「综合分析」等非文件来源。如果没有单一文件支撑，可以写「综合 multiple files」，但必须列出具体文件名。

示例：
> 根据侯亮总的 OKR 培训材料，OKR 的核心是聚焦最重要目标而非把所有目标都列出来[来源: OKR_101.pptx#第3页]。
> 这与北大分享版的观点一致，后者进一步强调了对齐的重要性[来源: 北大留学生版.pptx#第5页]。

回答尾部仍保留完整的引用列表：
> *来源：`<项目名>/<位置>/<文件名>`*

**文件读取时记录锚点：**
1. 读每份文件时，随手记录当前页码/段落号
2. 当你准备引用其中的某句话时，在草稿里标记来源
3. 最终输出时，把来源锚点内联到对应信息后面

### MECE 结构化输出（强制执行）

**问题类型决定输出结构：**

| 问题类型 | 输出格式 |
|---------|---------|
| 对比类（A 和 B 的区别/异同） | **对比表** — 横向维度，纵向双方 |
| 分类类（有哪几类/几种） | **编号列表 + 每类说明** — 金字塔结构 |
| 流程类（怎么做/步骤） | **编号步骤 + 子项说明** |
| 罗列类（有哪些文件/内容） | **表格** — 文件名/类型/关键内容 |
| 综合类（请分析/评价） | **先结论 → 后论据** 的金字塔结构 |

**基本原则：**
1. 检索类问题**禁止**仅用纯段落回答。至少使用一种结构化格式
2. 对比类问题**必须**使用对比表，**禁止**分别写两段描述
3. 每条结构化信息后同样需要附来源锚点 `[来源: 文件名#页码]`
4. 输出中优先用 markdown 表格/列表，避免散文段落

### 描述进化（每读完一个文件后执行）

```
1. 从刚读到的内容中提取 1-3 个核心短语
   （浓缩的关键词短句，如「氢能补贴标准2026」「IVC立项流程」）

2. 对比现有描述：
   → 这些短语是否已在描述中体现？
      - 已有 → 不做改动（去重）
      - 有新信息 → 追加到描述末尾（逗号分隔）

3. 长度控制：
   → 描述超过 ~100 字 → 合并精简旧部分，保留最新/最相关的
   → 无需精确字符串匹配，AI 自行判断去重和整合

4. 更新 data_structure.md 对应行

**[自检] 刚读完的文件，描述列更新了吗？打开 data_structure.md 确认。**

⚠️ 追加短语注意：data_structure.md 是 Markdown 表格，描述列中
禁止使用管道符 `|`，以免破坏表格结构。用逗号分隔多个短语。
```
